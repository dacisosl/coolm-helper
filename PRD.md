# PRD — 쿨메신저 일정 도우미

최종 수정: 2026-07-18 · 상태: M0(사전 조사) 완료, M1(파서) 착수 전

---

## 1. 배경과 목표

교사는 하루에도 수십 건의 쿨메신저 쪽지를 받고, 그 안의 일정(회의, 심의, 제출 기한)을
수첩·캘린더에 손으로 옮긴다. 이 과정에서 누락이 생기고, 캡처·복붙을 쓰면
학생 개인정보가 의도치 않게 온라인 서비스로 흘러갈 위험이 있다.

**목표:** 버튼 한 번으로 최근 쪽지에서 일정을 뽑아 확인 후 등록한다.
**비목표:** 백그라운드 감시, 자동 등록(확인 없는 등록), 메시지 원문의 외부 전송.

## 2. 사용자와 사용 시나리오

- 대상: 쿨메신저를 쓰는 초·중·고 교사 (1차: 개발자 본인, 2차: 동료 교사 배포)
- 시나리오: 쪽지 확인 후 "아 이거 일정이네" 싶을 때 플로팅 위젯의 [일정 등록] 클릭
  → 최근 쪽지에서 추출된 일정 후보 카드 확인 → 필요하면 수정 → [등록]

## 3. 전체 아키텍처 — 두 개의 존(Zone)

이 프로그램의 모든 설계는 **"로컬 존"과 "온라인 존"을 물리적으로 분리**하는 데서 출발한다.

```
┌─────────────────────────────── 로컬 존 (내 PC) ───────────────────────────────┐
│                                                                              │
│  쿨메신저 DB          parser/                     ui/              store/     │
│  ┌──────────┐   ┌─────────────────┐   ┌───────────────────┐   ┌──────────┐  │
│  │ *.udb    │──▶│ ① db_reader     │──▶│ ④ 미리보기 카드    │──▶│ 일정.json │  │
│  │ (SQLite) │복사│ ② date_parser   │   │   (편집·확인)      │   │ 할일.json │  │
│  └──────────┘   │ ③ pii_detector  │   │ ⑤ 내장 캘린더뷰    │   └──────────┘  │
│                 └─────────────────┘   └───────────────────┘                  │
│  students.txt ──▶ (③에서 사전으로 사용)          │                            │
│  tbl_member  ──▶ (③에서 사전으로 사용)          │ 사용자가 [구글에도 등록]     │
│                                                │ 체크했을 때만               │
└────────────────────────────────────────────────┼─────────────────────────────┘
                                                 ▼
                              ┌─── 온라인 존 (선택) ────────────┐
                              │  calendar_sync/                │
                              │  마스킹·확인된 제목 + 일시만     │
                              │  → Google Calendar API         │
                              └────────────────────────────────┘
```

**경계를 넘는 데이터는 딱 3가지뿐이다: 마스킹된 제목, 시작일시, 종료일시.**
메시지 원문, 발신자, 첨부파일명, 명단은 구조적으로 `calendar_sync/` 모듈에 전달되지 않는다
(함수 시그니처 수준에서 차단 — 원문을 넘길 방법 자체가 없게 설계).

## 4. 데이터 소스 (M0 조사 결과 확정)

| 항목 | 내용 |
|---|---|
| 위치 | `%LOCALAPPDATA%\CoolMessenger\Memo\*.udb` |
| 형식 | 암호화 없는 표준 SQLite 3 (WAL 모드) |
| 선택 규칙 | 폴더 내 **가장 최근 수정된** .udb (구버전 파일이 공존할 수 있음) |
| 받은 쪽지 | `tbl_recv` — `Sender`, `ReceiveDate`, `Title`, `MessageText`(평문 본문) |
| 보낸 쪽지 | `tbl_send` (대칭 구조, 초기 버전에서는 미사용) |
| 교직원 명단 | `tbl_member` — ID·이름 (PII 탐지 사전으로 재활용) |
| 날짜 형식 | 문자열 `"2026/07/16 17:04:52 (목)"` — 전용 파싱 필요 |
| 주의 | `MessageBody`는 압축 형식이므로 무시하고 `MessageText`만 사용 |

**접근 규칙 (고정):** 쿨메신저가 DB를 잠그고 있으므로
`udb + -wal + -shm` 세 파일을 임시 폴더에 복사(공유 읽기 모드) → 복사본을 읽기 전용(`mode=ro`)으로 열기.
원본은 어떤 경우에도 쓰기 모드로 열지 않는다. 사용 후 복사본은 삭제한다.

**폴백 (Plan B):** 실행 시 `tbl_recv`와 필수 컬럼 존재를 검사하고, 실패하면(쿨메신저 업데이트로
스키마 변경 등) 엑셀 내보내기 파싱 모드로 안내한다.
내보내기 파일: `Documents\CoolMessenger Files\Received Files\coolmsg_*.xls`,
시트 [받은메시지/보낸메시지], 열 [보낸사람 | 제목 | 날짜/시간 | 내용 | 첨부파일] (M0에서 구조 확인됨).

## 5. 메시지 파싱 원리 (parser/)

파싱은 5단계 파이프라인이고, 전부 정규식 + 사전 기반이다. AI·인터넷을 쓰지 않는다.

```
쪽지 1건 ──▶ (1)정규화 ──▶ (2)날짜·시간 후보 추출 ──▶ (3)기준일 해석 ──▶ (4)제목 생성 ──▶ 일정 후보
```

### (1) 정규화
공백 정리, 전각→반각(`１４：００`→`14:00`), 괄호 요일 통일.

### (2) 날짜·시간 후보 추출 — 정규식 패턴 그룹
| 유형 | 예시 | 처리 |
|---|---|---|
| 절대 날짜 | `7월 21일`, `7/21`, `2026-07-21`, `7.21(화)` | 그대로 날짜화 |
| 상대 날짜 | `내일`, `모레`, `다음 주 화요일`, `금주 금요일` | 기준일에서 계산 |
| 시간 | `14:00`, `오후 2시`, `2시 30분`, `14시` | 24시간제로 통일 |
| 기간 | `7/21(월)~7/25(금)`, `21일부터 25일까지` | 시작·종료 이벤트 |
| 기한 | `~7/21까지 제출`, `7월 21일 마감` | 종일 일정 + [마감] 태그 |

### (3) 기준일 해석
"내일", "다음 주"의 기준은 **오늘이 아니라 쪽지의 `ReceiveDate`**다.
(지난주에 받은 쪽지의 "내일"은 지난주 기준이어야 하므로.)
연도가 생략된 날짜(`3/21`)는 ReceiveDate와 가장 가까운 미래로 해석한다.

### (4) 제목 생성
우선순위: ① 쪽지 `Title`에서 날짜 표현을 제거한 나머지 → ② 본문 첫 문장.
말줄임·조사 정리 후 미리보기에서 사용자가 자유롭게 고칠 수 있으므로 완벽할 필요 없음.

### (5) 신뢰도와 실패 처리
날짜는 있는데 시간이 없으면 "종일"로, 아무것도 못 찾으면 해당 쪽지는 후보에서 제외하되
목록에 "일정 없음"으로 표시해 사용자가 수동 추가할 수 있게 한다.

## 6. 개인정보 구분 아키텍처 — 무엇이 남고 무엇이 나가는가

### 구분의 기본 사고방식
"이 텍스트가 개인정보인가?"를 완벽히 판별하는 것은 불가능하다. 그래서 이 도구는
**3중 구조**로 접근한다:

```
1층: 구조적 차단   — 원문·발신자는 애초에 온라인 존으로 가는 통로가 없음 (§3)
2층: 자동 탐지     — 아래 규칙으로 잡히는 건 빨간 글씨로 표시 (자동 마스킹 없음)
3층: 사람 확인     — 미리보기에서 사용자가 편집·확인 후 [등록]을 눌러야만 전송
```

즉 자동 탐지(2층)는 "완벽한 필터"가 아니라 **사용자의 확인(3층)을 돕는 하이라이터**다.
탐지된 내용을 지울지는 사용자가 인라인 편집으로 결정한다(자동으로 지우지 않는다 —
사용자 결정, 2026-07-18). 설령 2·3층이 모두 놓쳐도 1층 덕분에 원문 전체가 새는 일은 없다.

### 자동 탐지 4종 규칙 (pii_detector)
| # | 규칙 | 방식 | 예시 |
|---|---|---|---|
| 1 | 전화번호 | 정규식 | `010-1234-5678`, `031)123-4567` |
| 2 | 주민등록번호 | 정규식 + 체크섬 | `990101-1******` 포함 부분 패턴 |
| 3 | 호칭 패턴 | 정규식 | `김○○ 학생`, `○○○님`, `○○ 학부모님` |
| 4 | 명단 대조 | 사전 매칭 | `students.txt`(학생) + `tbl_member`(교직원) |

- 탐지된 구간은 미리보기에서 **빨간 글씨 + 밑줄**로 표시된다. 원문은 그대로 보인다.
- 제목에 탐지 항목이 있으면 카드에 경고줄("개인정보로 보이는 부분: ...")이 뜬다.
- 사용자가 제목을 편집해 지우거나, 그대로 두고 등록할 수 있다 — 선택은 사용자 몫.
- 원문은 로컬 존 밖으로 나가지 않으므로, 미탐의 피해 범위는
  사용자가 등록한 "제목 한 줄" 이내로 제한된다.

### 예시
```
원문(로컬):   "3학년 김철수 학생 학폭위 심의가 7월 21일(화) 14시에 열립니다.
              담당: 이영희 선생님(010-1234-5678)"
탐지·표시:    김철수, 이영희, 010-1234-5678 이 빨간 글씨로 표시됨
미리보기:     "3학년 김철수 학생 학폭위 심의" ← 사용자가 "학폭위 심의"로 편집 가능
온라인 전송분: 사용자가 편집·확인한 제목 + 일시  ← 이게 전부
```

## 7. 기능 요구사항

### P1 — 일정 등록 (MVP)
- 플로팅 위젯의 [일정 등록] 클릭 → **가장 최근 쪽지 N개**(기본 10, config.json의 recent_count)에서 일정 후보 추출 — 날짜 기준이 아니라 개수 기준(공휴일·연휴 대응). 미리보기 창에서 10/50/100개로 즉시 확장 가능
- 후보 카드: 제목(마스킹 반영) / 일시 / 원본 쪽지 펼쳐보기(로컬 표시 전용)
- 인라인 편집(제목·일시), [등록] / [건너뛰기]
- [등록] → 로컬 `store/events.json` 저장

### P2 — 내장 캘린더뷰 (2026-07-18 재설계)
- 테두리 없는 플랫 달력. 날짜 셀 아래에는 **일정 개수 배지만** 표시
- 날짜 클릭 → 오른쪽 패널에 투두리스트 스타일 목록: **[중요도 칩] + 제목 + 시간**
- 항목 클릭 → **아코디언 상세보기**: 제목·일시·종일·중요도·메모를 인라인 편집,
  [저장]으로 반영, [삭제] 지원. 중요도(높음/보통/낮음)는 색상 칩으로 구분
- 별도 할일 탭은 없음(삭제됨) — 마감형 일정도 같은 목록에서 관리
- 오늘 일정 요약을 플로팅 위젯에 뱃지로 표시
- 데이터는 전부 로컬 JSON — 이 단계까지 온라인 존 코드가 전혀 실행되지 않음

### P2.5 — 설정 모달 (⚙ 아이콘)
- **계정**: 저장 모드 선택 — **로컬 모드(기본값) / 구글 연동 모드(옵트인)**.
  휴대폰에서 일정을 보고 싶은 사용자는 구글 연동 모드 사용.
  구글 연동 안내 열기 · 계정 연결 해제
- **데이터**: 기본 쪽지 개수(recent_count), 데이터 폴더 열기, 현재 버전,
  자동 업데이트 확인 토글, 지금 업데이트 확인
- **개인정보**: 학생 명단(students.txt) 상태·열기/만들기, 탐지 정책 안내
- **위젯**: 항상 위 표시, 투명도(50~100%)

### P3 — 구글 캘린더 연동 (선택 기능)
- 설정에서 켜는 옵트인. 켜면 등록 카드에 [구글에도 등록] 체크박스 등장
- OAuth 로그인은 사용자 브라우저에서 진행, 토큰은 로컬 저장
- 전송 데이터: 마스킹된 제목, 시작·종료 일시 (그 외 필드 전송 금지)

### P6 — ⚡ 간편 등록 (v0.8.0 구현 완료)
- 흐름: 쿨메신저에서 쪽지를 **보고 있는 상태**에서 펭귄 ⚡ 클릭 →
  **지금 화면에 떠 있는 그 쪽지가 자동으로 채워진 작은 모달** → 수정 → 등록.
- 핵심 메커니즘 (2026-07-19 실기기 검증): **Windows UI 자동화**(화면낭독기와
  같은 표준 기술, capture.py)로 열려 있는 쿨메신저 창의 제목(Edit)·본문
  (Document)을 읽기 전용으로 가져온다. 키 시뮬레이션·클립보드 조작 없음.
- 읽은 텍스트를 DB 쪽지와 대조(match_captured, 공백 무시·최소 길이 가드)해
  원본을 찾으면 수신시각 기준 파싱 + 등록 표시(source_ref) 연동.
- 일정이 여러 개 감지되면 콤보로 선택, 날짜가 없으면 직접 선택 안내.
- 폴백: 쪽지 창이 없으면 → [클립보드에서 가져오기](쪽지 외 텍스트도 일정화)
  → [전체 목록 열기]. 기존 2분할 창은 그대로 유지 — 간편 등록은 입구 추가.

### P4 — 안내문구 보정 (공개용 글 전용)
- 가정통신문 등 **공개 예정 글**의 문구를 클라우드 AI로 다듬는 별도 기능
- 쪽지 파이프라인과 코드·UI 완전 분리(별도 탭, parser/ 모듈 import 금지)
- 입력창에 사용자가 직접 붙여넣은 텍스트만 처리 — 쪽지 자동 불러오기 없음

- **설치파일:** PyInstaller로 단일 exe → Inno Setup으로 설치 마법사 제작.
  첫 실행 시 `%LOCALAPPDATA%\CoolMessenger\Memo` 자동 탐지, 없으면 경로 선택 UI.
- **구글 OAuth와 배포:** 구글 API를 여러 사람이 쓰려면 Cloud 프로젝트의 앱 검증
  또는 테스트 사용자 등록이 필요하다. 내장 캘린더뷰(P2)가 기본이므로
  **구글 설정 없이도 완전한 사용이 가능**하고, P3는 원하는 사람만 개별 안내로 활성화한다.
- **백신 오탐:** PyInstaller exe는 SmartScreen 경고가 뜰 수 있음 — 배포 문서에 안내 포함.
- **스키마 변경 리스크:** 쿨메신저 업데이트로 .udb 구조가 바뀌면 시작 시 검사에서 감지하고
  Plan B(엑셀 내보내기) 모드로 전환 안내 (§4 폴백).
- **자동 업데이트:** 시작 시 `update_url`(version.json)에서 버전을 확인, 새 버전이면
  "업데이트 후 재시작하시겠습니까?" → 설치파일 자동 다운로드·조용한 설치·재실행.
  서버로 나가는 것은 버전 요청뿐. update_url이 비어 있으면 통신하지 않음.
  배포 준비물: GitHub Releases에 Setup.exe 업로드 + version.json 공개 URL (updater.py 참고).
